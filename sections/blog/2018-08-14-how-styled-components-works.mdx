CSS-in-JS is getting more and more common in the modern front-end development and especially in the React community. [styled-components](https://www.styled-components.com) stands out from [the list](https://github.com/tuchk4/awesome-css-in-js#libraries) because it is powered by [tagged templates](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Template_literals#Tagged_templates) and allows to create normal react components by only defining styles. It also solves such important problem like CSS modularity, provides non-CSS features like nesting, and all these features supplied with zero-configuration. Developers no need to think about uniq names for CSS classes they even don’t need to think about classes at all. But how is this power achieved?

***Note****: If you are not familiar with styled-components read *[*the official documentation*](https://www.styled-components.com/docs)* first.*

styled-components renders an element with 3 class names:

- this.props.className — optional passed by parent component.
- componentId — uniq identifier of a component but not component instance. This class has no any CSS rules but it is used in nesting selectors when need to [refer to other component](https://www.styled-components.com/docs/advanced#referring-to-other-components).

#### componentWillReceiveProps()

Now let’s try to change our button props when it is mounted. To do this we need to make more interactive example for our Button:
[**Live version**](https://jsfiddle.net/gevgeny/0ezu72Ly/)

Every time you click the button, the componentWillReceiveProps() is called with incremented sizeValue prop and does the same actions as the componentWillMount() does:

- Evaluate the tagged template.
- Generate the new CSS class name.
- Preprocess the styles with [stylis](https://github.com/thysultan/stylis.js).
- Inject the preprocessed CSS into the page.
If you check generated styles in dev tools after a few clicks you will see:
Yes, the only difference for each CSS class is font-size property and unused CSS classes are not removed. But why? Just because removing them adds performance overhead while keeping does not ([see Max’s Stoiber comment on this](https://github.com/styled-components/styled-components/issues/1431#issuecomment-358097912)).

There is one small optimization here: components without interpolations in the style string are marked as isStatic and this flag is checked in componentWillReceiveProps() in order to skip unnecessary calculations of the same styles.

### Performance tips

Knowing how styled-components works under the hood we can focus on performance better.

There is an easter egg in [the example with button](https://jsfiddle.net/gevgeny/0ezu72Ly/) (Hint: try to click the button more then 200 times and you will see the hidden message from styled-components in the console. No kidding!* 😉).

If you are too eager here is the message:

```js
Over 200 classes were generated for component styled.button.
Consider using the attrs method, together with a style object for frequently changed styles.

Example:
  const Component = styled.div.attrs({
    style: ({ background }) => ({
      background,
    }),
  })`width: 100%;`
```

Here is how the Button looks like after the refactoring:
*But should you use this technique for all you dynamic styles ? No. But my personal rule is to **use the style attr for all dynamic styles with undermined number of results**. For example if you have a component with customizable font-size like a [word cloud](https://www.jasondavies.com/wordcloud/) or list of tags loaded from a server with different colors it is better to use the style attr. But if you have variety of buttons like default, primary, warn and etc. in one component it is ok to use interpolation with conditions in the styles string.

In the examples below I use development version but in a production bundle you should always use the production build of styled-components because it is faster. Just like in React styled-components production build disables many dev warning but most importantly, it uses[CSSStyleSheet.insertRule()](https://developer.mozilla.org/en-US/docs/Web/API/CSSStyleSheet/insertRule) to inject generated styles in a page while development version uses [Node.appendChild()](https://developer.mozilla.org/en-US/docs/Web/API/Node/appendChild) ([Here Evan Scott shows](https://medium.com/styled-components/v3-1-0-such-perf-wow-many-streams-c45c434dbd03) how really fast insertRule is).

Also consider using [babel-plugin-styled-components](https://www.styled-components.com/docs/tooling#babel-plugin). It can minify or even preprocess styles before loading.

### Conclusion

styled-components workflow is pretty straightforward, it creates necessary CSS on the fly right before components render and it is fast enough despite evaluating tagged strings and preprocessing CSS right in the browser.
