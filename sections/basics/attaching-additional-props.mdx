## Attaching additional props

To avoid unnecessary wrappers that just pass on some props to the rendered component or element, you can use the [`.attrs` constructor](/docs/api#attrs). It allows you to attach additional props (or "attributes") to a component.

This way you can for example attach static props to an element or pass a third-party prop like `data-testid` to a component. Furthermore, you can also attach more dynamic props to a component. The `.attrs` object also takes functions, that receive the props that the component receives. The return value will be merged into the resulting props as well.

Here we render an `Input` component and attach some dynamic and static attributes to it:

```react
const Input = styled.input.attrs<{ $size?: string; }>(props => ({
  // we can define static props
  type: "text",

  // or we can define dynamic ones
  $size: props.$size || "1em",
}))`
  color: #BF4F74;
  font-size: 1em;
  border: 2px solid #BF4F74;
  border-radius: 3px;

  /* here we use the dynamically computed prop */
  margin: ${props => props.$size};
  padding: ${props => props.$size};
`;

render(
  <div>
    <Input placeholder="A small text input" />
    <br />
    <Input placeholder="A bigger text input" $size="2em" />
  </div>
);
```

As you can see, we get access to our newly created props in the interpolations, and the `type` attribute is passed down to the element.

### Overriding .attrs
Notice that when wrapping styled components, `.attrs` are applied from the innermost styled component to the outermost styled component.

This allows each wrapper to override nested uses of `.attrs`, similarly to how CSS properties defined later in a stylesheet override previous declarations.

`Input`'s `.attrs` are applied first, and then `PasswordInput`'s `.attrs`:
```react
const Input = styled.input.attrs<{ $size?: string }>((props) => ({
	type: 'text',
	$size: props.$size || '1em',
}))`
	border: 2px solid #bf4f74;
	margin: ${(props) => props.$size};
	padding: ${(props) => props.$size};
`;

// Input's attrs will be applied first, and then this attrs obj
const PasswordInput = styled(Input).attrs({
  type: "password",
})`
  // similarly, border will override Input's border
  border: 2px solid aqua;
`;

render(
  <div>
    <Input placeholder="A bigger text input" $size="2em" />
    <br />
    {/* Notice we can still use the size attr from Input */}
    <PasswordInput placeholder="A bigger password input" $size="2em" />
  </div>
);
```
This is why `PasswordInput` is of a `'password'` type, but still uses the `$size` attribute from `Input`.

### Props supplied via `.attrs` are optional | v6.4+

Any prop that `.attrs` provides a value for is automatically made optional on the resulting component's type. You no longer need to pass props that attrs already defaults:

```tsx
const Link = styled.a.attrs({ href: '#', rel: 'noopener' })`
  color: hotpink;
`;

// Both work — attrs provides the defaults
<Link>Click</Link>
<Link href="/about">About</Link>
```

Previously, TypeScript required you to pass `href` even though attrs supplied it.

### Bridging styles into props | v7+

Some third-party components — most notably `react-native-svg`'s `<Path>` and `<Circle>`, charting libraries, `<Image tintColor>`, icon libraries that take a `color` prop — style through component props instead of the `style` prop. Authoring those as CSS would normally be impossible. The function form of `.attrs` accepts a second `ast` argument that lets you read a value out of the rendered style and rewrite it onto the component as a prop:

```tsx
import { Path } from 'react-native-svg';

// Author the icon with CSS, render as <Path fill="red" stroke="black" />
const Icon = styled(Path).attrs((_props, ast) => ({
  fill: ast.pop('color'),
  stroke: ast.pop('borderColor', 'black'),
}))`
  color: red;
  border-color: navy;
`;
```

`ast.pop(key)` reads the value and removes the declaration's effect on the rendered style — useful when the prop is the *only* surface that should receive the value. `ast.peek(key)` reads without removing — useful when the prop and the CSS declaration should both flow through. Both accept an optional fallback as the second argument.

The first argument dispatches on shape. A CSS property name reads a declaration; a dot-separated path reads from the active theme:

```tsx
const Logo = styled(Image).attrs((_props, ast) => ({
  tintColor: ast.pop('palette.brand.primary'),
}))`
  width: 32px;
  height: 32px;
`;
```

Theme paths get autocomplete and value-type inference from your augmented `DefaultTheme`. Path keys are walked by JS property access, so any key shape works (`palette.brand.primary`, `colors.red-500`, `space.0`, …).

Callbacks that only read static declarations and theme paths are evaluated once at construction; renders pay nothing extra.
